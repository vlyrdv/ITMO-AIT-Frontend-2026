# Лабораторная работа 3: пояснение по выбору технологии

В рамках обучения в школе Movavi я познакомился как с **Vue.js**, так и с **Handlebars (HBS)**.  
Во время изучения этих инструментов я пришёл к выводу, что для моего проекта более подходящим решением оказался именно **HBS**, так как он лучше соответствует поставленной задаче и проще в использовании для небольших и средних по объёму приложений.

## Почему был выбран HBS вместо Vue

Хотя Vue.js является мощным инструментом для разработки SPA-приложений, в данном проекте его использование показалось мне избыточным.  
Мой проект не требовал сложной клиентской логики, большого количества интерактивных компонентов или полноценной клиентской маршрутизации. Основной упор был сделан на:

- отображение страниц;
- вывод данных с сервера;
- обработку форм;
- работу с каталогом, корзиной, заказами и авторизацией.

Для таких задач **Handlebars** оказался более удобным, так как:

- он проще в освоении и использовании;
- позволяет быстро собирать HTML-страницы из шаблонов;
- хорошо подходит для серверного рендеринга;
- не требует отдельной сложной клиентской архитектуры;
- позволяет переиспользовать части интерфейса через partials.

Именно поэтому в своём проекте я осознанно выбрал **HBS** и использовал его как основной инструмент для построения пользовательского интерфейса.

## Как HBS использовался в проекте

В проекте Handlebars применялся как шаблонизатор для серверного рендеринга страниц в связке с **Node.js + Express**.  
Сервер обрабатывал запросы пользователя, получал необходимые данные и передавал их в шаблоны `.hbs`, после чего пользователю возвращалась уже готовая HTML-страница.

### 1. Рендеринг страниц через Express

Для каждой страницы на сервере были настроены маршруты, которые возвращали соответствующий `.hbs`-шаблон.

Пример:

```js
response.render("index.hbs", {
  title: "Главная страница",
  islogin: req.session.islogin
});
```

Таким образом, шаблоны формировались на сервере, а пользователь сразу получал готовую страницу.

### 2. Использование отдельных шаблонов страниц

Для разных разделов сайта были созданы отдельные `.hbs`-файлы, например:

- главная страница;
- каталог;
- поиск;
- корзина;
- авторизация;
- заказы.

Это позволило логично разделить структуру проекта и упростить поддержку кода.

### 3. Использование partials

Повторяющиеся части интерфейса были вынесены в **partials** — например:

- `header`
- `footer`

Это позволило не дублировать один и тот же код на каждой странице и централизованно переиспользовать общие блоки интерфейса.

Пример подключения partial:

```hbs
{{> header}}
```

### 4. Вывод данных в шаблонах

HBS использовался для вывода данных, которые сервер передавал в шаблон: списков товаров, данных корзины, заказов, состояния авторизации и т.д.

Пример вывода списка:

```hbs
{{#each data}}
  <div class="product-card">
    <div class="name">{{this.product_name}}</div>
    <div class="price">{{this.price}} ₽</div>
  </div>
{{/each}}
```

Таким образом, данные из базы данных или серверной логики автоматически отображались на страницах.

### 5. Условный рендеринг

С помощью HBS в проекте также использовался условный вывод элементов интерфейса.  
Например, в зависимости от того, авторизован пользователь или нет, показывались разные ссылки в шапке сайта.

Пример:

```hbs
{{#if islogin}}
  <a href="/orders">{{islogin}}</a> - <a href="/logout">Выйти</a>
{{else}}
  <a href="/auth">Войти</a>
{{/if}}
```

Это позволило сделать интерфейс более удобным и динамичным без необходимости писать сложную клиентскую логику.

## Итог

Несмотря на то, что в рамках обучения я изучал и **Vue.js**, и **HBS**, для данного проекта я выбрал именно **Handlebars**, так как он оказался более простым, понятным и практичным инструментом для решения поставленных задач.  
С помощью HBS я реализовал серверный рендеринг страниц, переиспользуемые шаблоны, вывод данных и условное отображение элементов интерфейса. Для небольшого проекта такой подход оказался удобным и достаточным.
